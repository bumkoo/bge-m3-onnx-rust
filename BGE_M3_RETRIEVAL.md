# BGE-M3 Retrieval 방식 정리

## 개요

BGE-M3는 하나의 모델에서 세 가지 Retrieval 방식을 동시에 지원한다.
M3는 **Multi-Functionality, Multi-Linguality, Multi-Granularity**의 약자.

```
ONNX 추론 1회 실행
    │
    ├─ dense_vecs   [batch, 1024]        → Dense Retrieval
    ├─ sparse_vecs  [batch, token, 1]    → Sparse (Lexical) Retrieval
    └─ colbert_vecs [batch, token, 1024] → Multi-Vector (ColBERT) Retrieval
```

---

## 1. Dense Retrieval

### 개념

문장 전체를 **벡터 1개(1024차원)**로 압축하여 비교하는 방식.
의미적 유사성(semantic similarity)을 측정한다.

### 동작 과정

```
입력: "장삼풍의 태극권"
    ↓ Transformer 인코딩
    ↓ [CLS] 토큰의 벡터 추출 (CLS Pooling)
    ↓ L2 정규화
출력: [0.023, -0.041, 0.018, ...] ← float 1024개, 벡터 1개
```

gpahal 모델(`model_quantized.onnx`)은 CLS 풀링과 정규화가 모델 안에 포함되어 있어
`dense_vecs` 출력을 그대로 사용하면 된다.

### 유사도 계산

코사인 유사도(Cosine Similarity)를 사용한다.

```
              A · B         Σ(ai × bi)
cos(A, B) = ───── = ─────────────────────
             |A||B|   √Σ(ai²) × √Σ(bi²)

결과 범위: -1 ~ 1 (1에 가까울수록 유사)
```

### 강점

- 의미적 유사성을 잡아냄 ("무공의 비결" → "무술의 핵심 원리" 매칭 가능)
- 벡터 1개라 저장 공간 효율적 (문서당 약 4KB)
- ANN(Approximate Nearest Neighbor) 인덱스로 대규모 검색 가능

### 약점

- 문장 전체를 벡터 1개로 압축하므로 세부 정보 손실
- 정확한 키워드 매칭에 약함 ("청강만리" 같은 고유명사를 유사한 다른 단어로 대체 가능)

### 저장

```
LanceDB 테이블:
  doc_id | dense_vector [f32; 1024] | text
  1      | [0.023, -0.041, ...]     | "장삼풍이 무당산에서..."
```

### 실행 결과 예시

```
쿼리: "장삼풍의 태극권"
  1위: 0.6967 | 장삼풍이 무당산에서 태극권을 창시했다
  2위: 0.5176 | 장삼풍은 소림사에서 무당파를 세웠다
  3위: 0.4571 | 태극검법은 무당파의 진산지보이다
  5위: 0.3357 | 옥교룡이 이모백에게 청명보검을 건넸다 ← 관련 없어도 0.34
```

## 2. Sparse (Lexical) Retrieval

### 개념

전통적인 키워드 검색(BM25)의 **학습된 버전**.
각 토큰에 "이 문맥에서 얼마나 중요한가"라는 가중치를 매긴다.
대부분의 토큰 ID(25만개 어휘 중)는 0이고, 입력에 등장한 토큰만 값이 있어서 "Sparse(희소)"하다.

### 동작 과정

```
입력: "장삼풍의 태극권"
    ↓ Transformer 인코딩
    ↓ 각 토큰의 hidden state에 sparse_linear(1024→1) 적용
    ↓ ReLU (음수 → 0으로 제거)
출력: {▁장: 0.18, 삼: 0.17, 풍: 0.23, 의: 0.06, ▁태: 0.06, 극: 0.19, 권: 0.17}
      ← 토큰별 스칼라 가중치, 가중치 > 0인 것만 저장
```

gpahal 모델은 `sparse_linear + ReLU`가 모델 안에 포함되어 있어
`sparse_vecs` 출력에서 바로 가중치를 읽을 수 있다.

### 특수 토큰 처리

`[CLS]`(id=0), `[PAD]`(id=1), `[EOS]`(id=2), `[UNK]`(id=3)은 제외한다.
같은 토큰이 문장에서 여러 번 등장하면 최대 가중치를 사용한다.

### 유사도 계산

공통 토큰의 가중치를 곱해서 합산(dot product)한다.

```
쿼리:  {▁장:0.18, 삼:0.17, 풍:0.23, 극:0.19, 권:0.17, ...}
문서1: {▁장:0.20, 삼:0.15, 풍:0.25, 극:...,  권:0.22, ...}
문서2: {옥:0.30,  교:0.20, 룡:0.25, ...}

쿼리 vs 문서1: 0.18×0.20 + 0.17×0.15 + 0.23×0.25 + ... = 0.1875
쿼리 vs 문서2: 겹치는 토큰 없음 = 0.0000
```

### BM25와의 차이

| 항목 | BM25 | Sparse Retrieval |
|-----|------|-----------------|
| 가중치 계산 | 단어 빈도(TF) × 역문서 빈도(IDF) | Transformer가 문맥을 이해하여 계산 |
| 문맥 인식 | 없음 ("풍"은 어디서든 같은 가중치) | 있음 ("장삼풍"의 "풍" ≠ "태풍"의 "풍") |
| 학습 | 통계 기반 (학습 불필요) | 모델 학습으로 가중치 결정 |

### 강점

- 정확한 키워드 매칭에 강함 (고유명사, 무공 이름, 인물 이름)
- 겹치는 토큰이 없으면 0점 → 무관한 문서를 확실히 걸러냄
- 저장 공간 매우 효율적 (0이 아닌 토큰만 저장)

### 약점

- BGE-M3의 토크나이저(XLM-RoBERTa)가 한국어를 글자 단위로 분해
  "장삼풍" → "▁장" + "삼" + "풍" (3개 토큰)
  따라서 "풍림화산"의 "풍"도 매칭될 수 있음
- 의미적 유사성을 잡지 못함 ("무공" ≠ "무술")

### 저장

```
LanceDB 테이블 (JSON 컬럼):
  doc_id | sparse_json                          | text
  1      | {"장":0.20, "삼":0.15, "풍":0.25...} | "장삼풍이 무당산에서..."
```

### 실행 결과 예시

```
쿼리: "장삼풍의 태극권"
  1위: 0.1875 | 장삼풍이 무당산에서 태극권을 창시했다 ← 핵심 토큰 다수 겹침
  2위: 0.1206 | 장삼풍은 소림사에서 무당파를 세웠다   ← "장삼풍" 겹침
  3위: 0.0544 | 태극검법은 무당파의 진산지보이다       ← "태극" 겹침
  5위: 0.0000 | 옥교룡이 이모백에게 청명보검을 건넸다  ← 겹치는 토큰 없음, 완전 0
```

## 3. Multi-Vector (ColBERT) Retrieval

### 개념

**Contextualized Late Interaction over BERT**의 약자.
Dense가 문장을 벡터 1개로 압축하는 것과 달리,
ColBERT는 **모든 토큰의 벡터를 그대로 유지**한 채 비교한다.
각 토큰 벡터는 Transformer의 self-attention을 통해 문맥이 반영되어 있다.

### 동작 과정

```
입력: "장삼풍의 태극권"
    ↓ Transformer 인코딩
    ↓ 각 토큰의 hidden state에 colbert_linear(1024→1024) 적용
    ↓ L2 정규화
    ↓ [CLS], [SEP] 특수 토큰 제외
출력: 7개의 1024차원 벡터 (토큰마다 1개씩)
      ▁장 → [0.01, 0.03, ...]
      삼  → [0.02, -0.01, ...]
      풍  → [-0.01, 0.04, ...]
      의  → [0.00, 0.02, ...]
      ▁태 → [0.03, 0.01, ...]
      극  → [0.01, 0.05, ...]
      권  → [0.02, 0.03, ...]
```

핵심: 각 토큰 벡터는 단순한 글자 의미가 아니라 **문맥이 반영**되어 있다.
"장삼풍"에서의 "풍"과 "태풍"에서의 "풍"은 서로 다른 벡터가 된다.

### 유사도 계산 (MaxSim)

쿼리의 각 토큰이 문서의 모든 토큰과 비교하여 **최대 유사도**를 찾고,
이를 평균내어 최종 점수를 산출한다.

```
쿼리 토큰 7개 × 문서 토큰 15개 = 105번의 코사인 유사도 계산

▁장 → 문서 15개 토큰 중 최대 유사도 = 0.95 (문서의 "▁장"과 매칭)
삼  → 문서 15개 토큰 중 최대 유사도 = 0.93 (문서의 "삼"과 매칭)
풍  → 문서 15개 토큰 중 최대 유사도 = 0.94 (문서의 "풍"과 매칭)
의  → 문서 15개 토큰 중 최대 유사도 = 0.60 (문서의 "이"와 약하게 매칭)
▁태 → 문서 15개 토큰 중 최대 유사도 = 0.91 (문서의 "▁태"와 매칭)
극  → 문서 15개 토큰 중 최대 유사도 = 0.92 (문서의 "극"과 매칭)
권  → 문서 15개 토큰 중 최대 유사도 = 0.93 (문서의 "권"과 매칭)

최종 점수 = 평균(0.95+0.93+0.94+0.60+0.91+0.92+0.93) / 7 = 0.8832
```

수식으로 표현하면:

```
MaxSim(Q, D) = (1/N) × Σ max cos(qi, dj)
                       i   j

N = 쿼리 토큰 수, qi = 쿼리 i번째 토큰 벡터, dj = 문서 j번째 토큰 벡터
```

### 강점

- 토큰 수준의 정밀한 매칭 (인물+사물 조합 검색에서 정밀도 높음)
- 문맥을 반영한 토큰 벡터라 Sparse보다 의미적 매칭도 가능
- Dense보다 세부 정보 손실이 적음

### 약점

- 저장 공간이 큼 (토큰 수 × 1024 × 4바이트)
  문서 토큰 30개 → 30 × 1024 × 4 = 약 120KB (Dense의 30배)
- 계산량이 큼 (쿼리 토큰 × 문서 토큰 번의 코사인 유사도)
  대규모 검색에 직접 사용하기 어려움

### 저장

ColBERT 벡터는 크기가 크므로 사전 저장하지 않고,
리랭킹 시점에 후보 문서만 실시간 인코딩하여 계산하는 것이 일반적이다.

```
문서 1000개 기준 저장 공간 비교:
  Dense:   1000 × 4KB   = 약 4MB
  Sparse:  1000 × 수백B = 약 수백KB
  ColBERT: 1000 × 120KB = 약 120MB ← 저장 비효율
```

### 실행 결과 예시

```
쿼리: "장삼풍의 태극권"
  1위: 0.7792 | 장삼풍이 무당산에서 태극권을 창시했다
  2위: 0.6030 | 장삼풍은 소림사에서 무당파를 세웠다
  3위: 0.4653 | 태극검법은 무당파의 진산지보이다
  5위: 0.2159 | 옥교룡이 이모백에게 청명보검을 건넸다

[CLS]/[SEP] 특수 토큰을 제외해야 점수가 부풀지 않는다.
특수 토큰은 모든 문장에 공통이라 유사도를 균일하게 상승시킴.
```

## 4. Hybrid Retrieval (2단계 파이프라인)

### 개념

세 가지 방식을 조합하여 각각의 약점을 보완하는 실전 검색 전략.
BGE-M3 논문에서 권장하는 방식이다.

### 2단계 파이프라인

```
전체 문서 DB (수천~수만 개)
    │
    ├─ [1단계] Dense 검색 → 상위 K개 후보
    ├─ [1단계] Sparse 검색 → 상위 K개 후보
    │
    └─ 합집합 (중복 제거) → 후보 K~2K개
         │
         └─ [2단계] ColBERT 리랭킹 → 최종 순위
```

### 1단계: Dense + Sparse 후보 선별 (빠름)

Dense와 Sparse를 **동시에** 실행하여 각각의 상위 K개를 합친다.
두 방식이 서로 다른 문서를 찾아올 수 있으므로 합집합을 사용한다.

- Dense가 잡는 것: 의미적으로 관련 있지만 키워드가 다른 문서
- Sparse가 잡는 것: 정확한 키워드(고유명사 등)가 포함된 문서

이 단계에서 벡터 1개 vs 1개 비교(Dense) 또는 sparse dot product이므로 빠르다.

### 2단계: ColBERT 리랭킹 (정밀)

1단계에서 선별된 소수의 후보만 ColBERT MaxSim으로 재비교한다.
전체 DB를 ColBERT로 비교하면 계산량이 폭발하지만,
후보 3~10개만 비교하면 충분히 감당 가능하다.

```
문서 1000개 기준 계산량 비교:

[방법 A] ColBERT 전수 조사:
  쿼리 토큰 7개 × 문서 토큰 평균 30개 × 1000개 = 210,000번 코사인 유사도

[방법 B] 2단계 파이프라인:
  1단계: Dense 1000번 + Sparse 1000번 = 2000번 (빠름)
  2단계: ColBERT 5개 × 7 × 30 = 1,050번
  합계: 약 3,050번 ← 70배 절약
```

### 가중 합산 방식 (대안)

리랭킹 대신 세 가지 점수를 가중 합산하는 방식도 있다.

```
S_rank = w1 × S_dense + w2 × S_sparse + w3 × S_colbert

논문 권장 기본값: w1=0.4, w2=0.2, w3=0.4
시나리오에 따라 조절 가능
```

## 5. 세 가지 방식 비교 요약

| 항목 | Dense | Sparse | ColBERT |
|-----|-------|--------|---------|
| 출력 형태 | 벡터 1개 [1024] | {토큰: 가중치} | 벡터 N개 [N×1024] |
| 비교 방식 | 코사인 유사도 | Dot product | MaxSim |
| 비교 단위 | 문장 vs 문장 | 단어 vs 단어 | 토큰 vs 토큰 (문맥 포함) |
| 의미 유사성 | 강함 | 약함 | 강함 |
| 키워드 매칭 | 약함 | 강함 | 중간 |
| 검색 속도 | 빠름 | 빠름 | 느림 |
| 저장 공간 | 4KB/문서 | 수백B/문서 | 120KB/문서 |
| 적합한 용도 | 1단계 검색 | 1단계 검색 | 2단계 리랭킹 |

## 6. 칠국춘추 NPC 메모리 적용 계획

### 저장 구조

```
LanceDB "npc_memories" 테이블:
  doc_id | dense_vector [f32;1024] | sparse_json           | text           | npc_id | timestamp
  1      | [0.023, ...]           | {"장":0.20, "풍":0.25} | "장삼풍이..."  | npc_01 | 1700000000
```

- Dense: LanceDB ANN 인덱스로 벡터 검색
- Sparse: JSON 컬럼으로 저장, Rust에서 dot product 계산
- ColBERT: 저장하지 않고 리랭킹 시점에 실시간 계산

### 검색 시나리오 예시

```
플레이어: "옥교룡이 청명보검을 어디서 얻었어?"

[1단계 - Dense + Sparse 동시 검색]
  Dense  → 의미적으로 "보검 획득" 관련 기억 상위 5개
  Sparse → "옥교룡", "청명보검" 토큰이 포함된 기억 상위 5개
  합집합 → 약 7개 후보

[2단계 - ColBERT 리랭킹]
  7개 후보만 MaxSim 계산 → 최종 상위 3개

[3단계 - LLM 응답 생성]
  상위 3개 기억을 프롬프트에 포함하여 NPC 대사 생성
```

### 임베딩 시점

```
ONNX 추론 1회 실행 (문서 저장 시)
  → dense_vecs   → DB에 저장 ✅
  → sparse_vecs  → DB에 저장 ✅
  → colbert_vecs → 저장하지 않음 ❌ (리랭킹 시 재계산)

ONNX 추론 1회 실행 (쿼리 검색 시)
  → dense_vecs   → DB의 Dense와 비교
  → sparse_vecs  → DB의 Sparse와 비교
  → colbert_vecs → 후보 문서만 재인코딩하여 MaxSim
```
